# README 작성 가이드라인

> Teamitaka 프로젝트의 README.md를 효과적으로 작성하고 유지보수하기 위한 가이드

## 📝 README 작성 철학

### 목적
README는 프로젝트의 **첫인상**이자 **공식 문서**입니다. 다음 목적을 달성해야 합니다:

1. **신규 팀원 온보딩**: 30분 안에 프로젝트를 이해하고 개발 환경을 구축할 수 있어야 함
2. **기능 파악**: 현재 구현된 기능과 진행 중인 작업을 명확히 전달
3. **기술 스택 소개**: 사용된 기술과 선택 이유를 설명
4. **문제 해결**: 자주 발생하는 문제와 해결 방법 제공
5. **프로젝트 현황**: 브랜치별 작업 상태와 개발 진행도 공유

### 대상 독자

| 독자 | 필요 정보 | 우선순위 |
|------|-----------|----------|
| 신규 개발자 | 빠른 시작, 환경 설정, 개발 가이드 | ⭐⭐⭐ |
| 기존 팀원 | 브랜치 현황, API 문서, 문제 해결 | ⭐⭐⭐ |
| PM/기획자 | 주요 기능, 배포 상태, 브랜치 현황 | ⭐⭐ |
| 외부 기여자 | Git 워크플로우, 컨벤션, 라이선스 | ⭐ |

### 핵심 원칙

**1. 명확성 (Clarity)**
- 전문 용어는 설명과 함께 사용
- 각 섹션의 목적이 명확해야 함
- 애매한 표현 금지 ("아마도", "~인 것 같음" 등)

**2. 완결성 (Completeness)**
- 프로젝트 실행에 필요한 모든 정보 포함
- 외부 문서는 링크로 연결
- 예외 상황과 문제 해결 방법 제공

**3. 유지보수성 (Maintainability)**
- 변경 가능성이 높은 정보는 자동화하거나 별도 파일로 분리
- 업데이트 날짜 명시
- 구조화된 형식 사용 (표, 리스트 등)

**4. 접근성 (Accessibility)**
- 이모지로 시각적 구분
- 목차 제공 (긴 문서의 경우)
- 코드 블록에 언어 명시

## 🎯 필수 섹션 작성 가이드

### 1. 프로젝트 제목 및 소개

**목적**: 3초 안에 프로젝트가 무엇인지 이해시키기

**포함 요소**:
- 프로젝트명 (정확한 표기)
- 한 줄 설명 (서비스명 + 핵심 가치)
- 간단한 설명 (2-3문장)

**좋은 예시**:
```markdown
# Teamitaka Frontend

> 티미타카 - 팀 프로젝트 매칭·관리·평가 플랫폼

React 기반 프론트엔드 애플리케이션으로, 팀 프로젝트의 전체 라이프사이클(모집 → 매칭 → 관리 → 평가)을 지원합니다.
```

**나쁜 예시**:
```markdown
# TeamItaka

프론트엔드 프로젝트입니다.
```
❌ 서비스명 오타, 핵심 가치 부재, 설명 불충분

### 2. 주요 기능

**목적**: 구현된 기능을 카테고리별로 명확히 전달

**작성 방법**:
- 이모지로 카테고리 구분
- 기능명을 **볼드체**로 강조
- 각 기능은 한 줄로 간결하게 설명
- 구현 완료된 기능만 작성 (계획 중인 기능은 별도 섹션)

**좋은 예시**:
```markdown
### 🎯 프로젝트 모집 & 매칭
- **모집글 작성**: 3단계 플로우로 손쉬운 프로젝트 모집글 작성
- **지원 관리**: 지원자 확인 및 승인/거절
- **팀 매칭**: 관심사 기반 팀원 추천 및 매칭
```

**나쁜 예시**:
```markdown
## 기능
- 프로젝트를 만들 수 있습니다
- 여러 가지 기능이 있습니다
- 앞으로 추가될 기능도 많습니다
```
❌ 구체성 부족, 카테고리 없음, 미래 기능 혼재

### 3. 기술 스택

**목적**: 사용된 기술과 버전 정보 제공

**작성 방법**:
- 카테고리별로 분류 (Core, Styling, UI Libraries 등)
- 주요 기술은 **볼드체**로 강조
- 버전 정보 포함
- 선택 이유를 간단히 설명 (선택 사항)

**좋은 예시**:
```markdown
### Core
- **React 18**: 최신 React 기능 활용
- **React Router DOM 7**: 선언적 라우팅
- **Redux Toolkit**: 상태 관리

### Styling
- **SASS (SCSS)**: 모듈화된 스타일링
- **반응형 디자인**: 모바일/태블릿/데스크톱 지원
```

**나쁜 예시**:
```markdown
## 기술
- React
- CSS
- JavaScript
```
❌ 버전 정보 없음, 카테고리 없음, 설명 부족

### 4. 빠른 시작 (Quick Start)

**목적**: 신규 개발자가 5분 안에 프로젝트를 실행할 수 있도록 안내

**작성 방법**:
- 단계별로 번호 매기기
- 각 단계마다 명령어와 설명 제공
- 예상 결과 명시
- 실패 시 대처 방법 링크

**좋은 예시**:
```markdown
### 1. 의존성 설치

\`\`\`bash
npm ci
\`\`\`

### 2. 환경변수 설정

\`\`\`bash
# .env.example을 .env.local로 복사
cp .env.example .env.local
\`\`\`

기본값으로 배포된 백엔드 서버가 설정됩니다.

### 3. 개발 서버 실행

\`\`\`bash
npm start
\`\`\`

- 기본 주소: `http://localhost:3000`
- 화면 상단에 **개발 모드 배너**가 표시되면 정상 작동
```

**나쁜 예시**:
```markdown
## 설치
npm install 하고 npm start 하면 됩니다.
```
❌ 구체적 단계 없음, 환경 설정 누락, 확인 방법 없음

**작성 방법**:

- import 문 포함
- 주석으로 설명 추가
- 성공/실패 케이스 모두 다루기

### 모집글 관리

  project_type: 'side',

  recruitment_end: '2025-02-20'

// 모집글 조회

### 6. 브랜치 현황

**목적**: 현재 진행 중인 작업과 브랜치 상태 공유

**업데이트 주기**: 주 1회 또는 주요 변경 시

**작성 방법**:
- 표 형식으로 정리
- 상태는 이모지로 시각화
- 최근 업데이트 날짜 명시
- 담당자 태그 (@username)

**좋은 예시**:
```markdown
### 활성 개발 브랜치

| 브랜치 | 담당자 | 작업 내용 | 상태 | 최근 업데이트 |
|--------|--------|-----------|------|---------------|
| `woo` | @woo | 모집글 백엔드 API 연동 | 🚀 진행중 | 2025-11-16 |
| `yeye` | @yeye | 날짜 & 아이콘 컨텍스트 | 🔄 진행중 | 2025-11-09 |
```

**업데이트 시기**:
- ✅ 새 브랜치 생성 시
- ✅ 브랜치 병합 시
- ✅ 작업 상태 변경 시 (진행중 → 완료)
- ✅ 주간 회의 후

### 7. 문제 해결 (FAQ)

**목적**: 자주 발생하는 문제와 해결 방법 제공

**작성 방법**:
- 증상 → 해결 구조
- 실제 에러 메시지 포함
- 단계별 해결 방법
- 관련 문서 링크

**좋은 예시**:
```markdown
### CORS 오류
**증상**: `Access-Control-Allow-Origin` 에러
**해결**: 백엔드 Render 환경변수 `CORS_ORIGIN`에 프론트엔드 도메인 추가

### 401 인증 오류
**증상**: API 호출 시 401 Unauthorized
**해결**:
- JWT 토큰 유효성 확인
- `localStorage`의 `authToken` 확인
- 토큰 만료 시 재로그인
```
